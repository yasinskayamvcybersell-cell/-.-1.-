# 14 Артефакт. Основные представления

## Спортивная социальная платформа

**Автор:** Архитектор ПО Ясинская М.В.

**Дата:** Июнь 2026  

**Версия:** 1.0

## 14.1. Функциональное представление

**Описание:**  
Функциональное представление показывает, как пользовательские действия (регистрация, запись тренировки, просмотр ленты, участие в челлендже, покупка) инициируют вызовы соответствующих микросервисов и генерацию доменных событий. Оно демонстрирует сквозной поток: от действий пользователя до фиксации результатов в системе. Это представление важно для понимания того, как бизнес-требования (артефакт 2) отображаются на архитектуру и какие события публикуются для дальнейшей обработки.

<img width="5713" height="1834" alt="Функциональная архитектура" src="https://github.com/user-attachments/assets/7886f14f-44b2-48aa-b0b6-3c5c3a83d3e3" />

**Пояснение к диаграмме:**
Каждое действие пользователя направляется в соответствующий сервис, который после обработки генерирует событие в шину (Kafka). Это позволяет асинхронно запускать последующие процессы (уведомления, обновление статистики, аналитику) без блокировки основного потока. Такая архитектура обеспечивает слабую связанность и масштабируемость.

## 14.2. Информационное представление (модели данных)

**Описание:**
Информационное представление описывает основные сущности предметной области и их связи. В системе выделены пять ключевых сущностей: Пользователь, Тренировка, Инвентарь, Группа и Челлендж. Модель данных построена на основе доменной модели (артефакт 4) и учитывает требования к хранению (артефакт 9) – транзакционность (PostgreSQL), временные ряды (TimescaleDB) и поиск (Elasticsearch). Диаграмма ER показывает атрибуты каждой сущности и связи между ними (например, пользователь создаёт тренировки, владеет инвентарём, принадлежит группам и участвует в челленджах).

<img width="4870" height="2643" alt="Модель данных основных сущностей" src="https://github.com/user-attachments/assets/51a3644d-dd54-47b3-b8d7-0df44c35f85a" />

**Пояснение:**
Эта модель является основой для проектирования схем БД. Для каждого сервиса выделены свои таблицы с соответствующими полями. Связи между сущностями реализованы через внешние ключи. Модель поддерживает расширение – добавление новых атрибутов или связей не требует изменения всей системы, что соответствует принципу расширяемости (артефакт 8).

## 14.3. Многозадачность (Concurrency)

**Описание:**
Представление многозадачности показывает, как система обрабатывает пиковые нагрузки при записи тренировок. Вместо синхронной обработки (которая может привести к задержкам и отказам) используется асинхронная модель через Kafka. Это позволяет распределить нагрузку между несколькими потребителями, каждый из которых выполняет свою задачу: обновление статистики, лидербордов, проверку инвентаря и отправку уведомлений. Данный подход решает требования масштабируемости и отказоустойчивости (артефакт 9).

<img width="4716" height="1635" alt="Concurrency – асинхронная обработка тренировок" src="https://github.com/user-attachments/assets/956a960e-0cf6-4ab6-9eb1-33e9434d9feb" />

**Пояснение:**
Как видно из диаграммы, после сохранения тренировки в TimescaleDB сервис публикует событие training_completed. Четыре независимых потребителя (возможно, запущенных в разных подах) параллельно обрабатывают это событие. Такой подход даёт следующие преимущества:
- **Масштабируемость** – каждый потребитель может быть горизонтально масштабирован независимо.
- **Отказоустойчивость** – если один потребитель упал, сообщение остаётся в очереди и будет обработано другим экземпляром.
- **Сглаживание пиков** – буферизация в Kafka позволяет системе принимать большое количество запросов, не блокируя пользовательский интерфейс.

## 14.4. Инфраструктурное представление

**Описание:**
Инфраструктурное представление описывает физическое развёртывание системы: два активных ЦОД в разных регионах (us-east-1 и eu-west-1) для обеспечения географической избыточности, а также резервный DRP-ЦОД для катастрофического восстановления. Показаны основные компоненты: глобальный балансировщик, API Gateway, Kubernetes-кластеры, базы данных (с репликацией), Kafka (с MirrorMaker) и системы мониторинга. Это представление реализует требования надёжности, отказоустойчивости и геораспределения (артефакты 8 и 9).

 <img width="8192" height="4276" alt="Инфраструктура" src="https://github.com/user-attachments/assets/17912a2c-8207-4a5c-9b30-55cd7c6812a8" />

**Пояснение:**
- **Global Load Balancer** распределяет трафик между регионами (active-active), автоматически отключая нездоровый ЦОД.
- **Kubernetes** обеспечивает оркестрацию и самовосстановление подов.
- **Базы данных** имеют синхронную репликацию (PostgreSQL) и асинхронную (TimescaleDB, Redis) для минимизации потерь данных при сбое.
- **Kafka MirrorMaker** реплицирует события между регионами для обеспечения согласованности.
- **DRP-ЦОД** содержит бэкапы и минимальный набор ресурсов для восстановления в случае полного отказа обоих основных ЦОД.

## 14.5. Представление безопасности

**Описание:**
Безопасность системы построена на многоуровневой модели, охватывающей транспортный уровень, аутентификацию, авторизацию и защиту данных. Это представление адресует требования защиты пользовательских данных (артефакт 9) и регуляторные ограничения (GDPR, 152-ФЗ). Каждый уровень включает конкретные механизмы, которые в совокупности обеспечивают комплексную защиту.

<img width="3218" height="2970" alt="Безопасность" src="https://github.com/user-attachments/assets/de39186e-6738-49d1-a5d8-2839059711cf" />

**Пояснение:**
- **Транспортный уровень** – все внешние и внутренние соединения используют TLS 1.3; между микросервисами применяется mTLS для взаимной аутентификации.
- **Аутентификация** – JWT с коротким сроком жизни (15 минут) и refresh-токенами; поддержка OAuth2 для входа через социальные сети; двухфакторная аутентификация для критических действий (изменение профиля, отмена заказа).
- **Авторизация** – ролевая модель (пользователь, тренер, администратор) с централизованным контролем через Open Policy Agent (OPA); все запросы к данным проверяются на соответствие политикам.
- **Защита данных** – персональные данные шифруются в БД (AES-256-GCM), чувствительные поля маскируются в логах, бэкапы также шифруются. Аудит доступа фиксирует все обращения к критическим данным для последующего расследования.
