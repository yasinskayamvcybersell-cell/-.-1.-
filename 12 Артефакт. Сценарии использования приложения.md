# 12 Артефакт. Сценарии использования приложения

## Спортивная социальная платформа

**Автор:** Архитектор ПО Ясинская М.В.  

**Дата:** Июнь 2026  

**Версия:** 1.0

### UC-01: Регистрация и онбординг

**Идентификатор:** UC-01  
**Название:** Регистрация и онбординг нового пользователя  
**Участники:** Новый пользователь  
**Предусловия:** Пользователь установил приложение, имеет доступ к интернету.

**Основной поток:**
1. Пользователь открывает приложение и нажимает "Зарегистрироваться".
2. Система отображает форму регистрации (email, пароль, подтверждение пароля) или опцию OAuth2.
3. Пользователь вводит данные. Пароль должен соответствовать политике сложности (≥8 символов, буквы и цифры).
4. Система проверяет уникальность email и создаёт учётную запись (пароль хешируется bcrypt).
5. Пользователь заполняет профиль: имя, вид спорта, уровень подготовки, локация.
6. Система предлагает подписку на популярные группы и челленджи.
7. Пользователь подтверждает подписки.
8. Система отправляет приветственное уведомление и показывает рекомендации тренировок.
9. Пользователь переходит на главный экран.

**Альтернативные потоки:**
- Email уже занят → сообщение об ошибке.
- Пароль слабый → предложение усложнить.
- OAuth2 неуспешен → предложение повторить или использовать email.

**Постусловия:** Учётная запись создана, профиль заполнен, подписки настроены. Зафиксировано событие `user_registered`.

### UC-02: Запись и сохранение тренировки

**Идентификатор:** UC-02  
**Название:** Запись и сохранение тренировки  
**Участники:** Зарегистрированный пользователь  
**Предусловия:** Пользователь авторизован, GPS/датчики активны.

**Основной поток:**
1. Пользователь нажимает "Начать тренировку".
2. Система запрашивает разрешения на GPS/датчики (если не даны).
3. Начинается запись данных: GPS-трек, пульс, шаги, время.
4. На экране отображается текущая статистика (дистанция, темп, пульс, длительность).
5. Пользователь завершает тренировку, нажимает "Сохранить".
6. Система показывает итоговую статистику, пользователь может добавить название/комментарий.
7. Пользователь подтверждает сохранение.
8. Данные отправляются в Workout Service, сохраняются в TimescaleDB, публикуется событие `training_completed`.
9. Асинхронные обработчики:
   - обновляют статистику пользователя;
   - проверяют инвентарь (пробег кроссовок);
   - обновляют лидерборды;
   - отправляют уведомления друзьям.
10. Пользователь видит подтверждение и может поделиться в ленте.

**Альтернативные потоки:**
- Потеря GPS → запись по шагомерам, уведомление.
- Ошибка сохранения → локальное сохранение и автоматический ретрай (до 3 раз).
- Нет интернета → локальное сохранение, отправка позже.

**Постусловия:** Тренировка сохранена, все связанные данные обновлены. Событие `training_completed` зафиксировано.

### UC-03: Просмотр ленты и взаимодействие

**Идентификатор:** UC-03  
**Название:** Просмотр ленты и социальное взаимодействие  
**Участники:** Зарегистрированный пользователь  
**Предусловия:** Пользователь авторизован.

**Основной поток:**
1. Пользователь открывает раздел "Лента".
2. Система загружает посты друзей, рекомендации, новости спорта.
3. Пользователь может:
   - поставить лайк;
   - оставить комментарий;
   - поделиться постом;
   - подписаться на автора;
   - перейти в профиль автора.
4. Действия отправляются в Social Service, сохраняются в PostgreSQL и индексируются в Elasticsearch.
5. Отображение обновляется (счётчики, комментарии).

**Альтернативные потоки:**
- Медленное соединение → показ кэшированной ленты.
- Пост удалён → сообщение "недоступен".
- Комментарий с запрещённым контентом → отклонён.

**Постусловия:** Взаимодействия сохранены. События `post_liked`, `comment_created`, `user_followed` зафиксированы.

### UC-04: Участие в челлендже

**Идентификатор:** UC-04  
**Название:** Участие в челлендже  
**Участники:** Зарегистрированный пользователь  
**Предусловия:** Пользователь авторизован, активный челлендж существует.

**Основной поток:**
1. Пользователь открывает раздел "Челленджи".
2. Система отображает список активных челленджей с условиями и сроками.
3. Пользователь выбирает челлендж, нажимает "Участвовать".
4. Пользователь добавляется в участники.
5. Тренировки автоматически учитываются в прогрессе.
6. Пользователь может видеть свой прогресс и место в лидерборде.
7. При достижении цели система выдаёт награды (очки, значки) и публикует событие `challenge_completed`.
8. Если цель не достигнута к сроку – уведомление о завершении без награды.

**Альтернативные потоки:**
- Уже участвует → сообщение "Вы уже участвуете".
- Челлендж завершён → невозможно вступить.
- Нет синхронизации данных → используется последний известный прогресс.

**Постусловия:** Участие зафиксировано, прогресс отслеживается. При успехе – награды выданы, событие зафиксировано.

### UC-05: Переход в магазин и покупка

**Идентификатор:** UC-05  
**Название:** Переход в интернет-магазин и совершение покупки  
**Участники:** Зарегистрированный пользователь  
**Предусловия:** Пользователь авторизован, есть интеграция с магазином.

**Основной поток:**
1. Пользователь получает рекомендацию товара (после тренировки, в ленте, в инвентаре).
2. Нажимает "Перейти в магазин".
3. Происходит единый вход в магазин (через JWT).
4. Пользователь выбирает товар, размер, цвет, добавляет в корзину.
5. Переходит к оформлению, указывает адрес доставки и способ оплаты.
6. Система резервирует товар (Inventory API) и инициирует платёж.
7. При успехе – заказ создаётся (Orders Service), публикуется событие `order_placed`.
8. Пользователь получает подтверждение и уведомление.
9. После доставки инвентарь (обувь/снаряды) автоматически добавляется в профиль.

**Альтернативные потоки:**
- Товар закончился → предложение аналога.
- Ошибка оплаты → сообщение, предложение другого метода.
- Сбой интеграции → сообщение о недоступности, сохранение контекста.

**Постусловия:** Заказ создан, платёж подтверждён. Инвентарь обновляется после доставки. Событие `order_placed` зафиксировано.
