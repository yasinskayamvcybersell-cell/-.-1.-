# 11 Артефакт. ADR – Архитектурные решения

## Спортивная социальная платформа

**Автор:** Архитектор ПО Ясинская М.В.

**Дата:** Июнь 2026  

**Версия:** 1.0

## Список архитектурных решений

В таблице ниже приведены ключевые архитектурные решения, сравнение альтернатив и обоснование выбора.

| Решение | Контекст | Выбранное решение | Последствия (плюсы / минусы) | Рассмотренные альтернативы | Почему выбрано |
|---------|----------|-------------------|-----------------------------|---------------------------|----------------|
| **1. Архитектурный стиль** | Команды распределенные, высокие нагрузки, частые обновления | Микросервисы (9 доменов) | + Независимое масштабирование и развертывание; + Командная автономия; - Усложнение эксплуатации; - Требуются саги | Монолит, SOA | Монолит не масштабируется, SOA менее гибкая. |
| **2. Взаимодействие сервисов** | Многие сценарии асинхронны, требуется низкая связанность | Событийная шина (Apache Kafka) | + Слабая связанность; + Легко добавлять новых потребителей; - Eventual consistency; - Управление схемами | Синхронные REST-вызовы, RabbitMQ | Синхронные вызовы повышают связанность, RabbitMQ уступает в пропускной способности. |
| **3. API для клиентов** | Три типа клиентов: мобильные/веб, сервисы, IoT | GraphQL (внешние), gRPC (внутренние), MQTT/WebSocket (IoT) | + GraphQL снижает число запросов; + gRPC высокопроизводителен; + MQTT лёгкий; - Сложность поддержки нескольких протоколов | Только REST, только gRPC | REST избыточен, gRPC сложен для браузеров. |
| **4. Хранение данных** | Разные требования: транзакции, временные ряды, поиск, кэш | PostgreSQL, TimescaleDB, Elasticsearch, Redis, S3 | + Каждый сервис использует оптимальное хранилище; - Усложнение управления; - Синхронизация между БД | Единая PostgreSQL, NoSQL для всего | Единая БД не справляется с поиском и временными рядами. |
| **5. Облачная инфраструктура** | Использование нескольких облаков, риск vendor lock-in | Multi-cloud + Kubernetes (AWS, GCP, Azure) | + Облачная агностичность; + Географическое резервирование; - Требуется K8s экспертиза; - Управление секретами сложнее | Единый провайдер | Единый провайдер создаёт зависимость. |
| **6. Распределённые транзакции** | Согласованность между сервисами (запись тренировки + инвентарь) | Saga + координатор (оркестратор) | + Согласованность без 2PC; + Отказоустойчивость; - Сложность разработки; - Задержки при компенсациях | 2PC, eventual consistency без саг | 2PC снижает доступность, eventual без саг недопустима для инвентаря. |
| **7. Аутентификация и авторизация** | Миллионы пользователей, интеграция с соцсетями | JWT + OAuth2 + RBAC + OPA | + Безопасное хранение; + Централизованное управление; - Управление ключами (Vault) | Сессионные cookie, Basic Auth | Cookie не масштабируются, Basic Auth небезопасен. |
| **8. Наблюдаемость** | Распределённая система, требуется быстрый поиск проблем | OpenTelemetry + ELK + Prometheus + Grafana + Jaeger | + Полная наблюдаемость; + Бизнес-метрики; - Высокая стоимость хранения; - Сложность настройки | Только логи, Splunk | Только логи недостаточны, Splunk дорог. |
