## Список архитектурных решений

### ADR-001: Микросервисная архитектура

**Контекст**  
Компания имеет распределённые команды разработчиков, использующих разные языки программирования. Система должна масштабироваться под десятки миллионов пользователей и поддерживать частые обновления.

**Решение**  
Выбрана **микросервисная архитектура** с выделением 9 доменов (User, Social, Workout, Planning, Gear, Gamification, Promotions, Notifications, Integrations).

**Последствия**  
- ✅ Независимое масштабирование и развертывание сервисов.  
- ✅ Команды могут использовать разные стеки технологий.  
- ❌ Усложняется эксплуатация (требуются оркестрация, наблюдаемость).  
- ❌ Необходимость управления распределёнными транзакциями (саги).

**Альтернативы**  
- Монолит – отброшен из-за сложности масштабирования и командной автономии.  
- SOA – отброшен из-за более низкой гибкости по сравнению с микросервисами.

### ADR-002: Событийно-ориентированная архитектура (Event-Driven)

**Контекст**  
Многие сценарии (отправка уведомлений, обновление лидербордов, интеграция с магазином) могут быть выполнены асинхронно. Требуется снизить связанность между сервисами.

**Решение**  
Внедрена **событийная шина (Apache Kafka)** для обмена сообщениями между сервисами. Все доменные события публикуются в Kafka и потребляются заинтересованными подписчиками.

**Последствия**  
- ✅ Слабая связанность – сервисы не вызывают друг друга синхронно.  
- ✅ Легко добавлять новых потребителей (аналитика, аудит).  
- ❌ Возникает проблема eventual consistency (необходимы саги).  
- ❌ Требуется управление схемами событий (Avro/JSON Schema).

**Альтернативы**  
- Синхронные REST-вызовы – отброшены из-за высокой связанности и проблем с отказоустойчивостью.  
- RabbitMQ – отброшен из-за меньшей пропускной способности и ограниченных возможностей репликации.

### ADR-003: Выбор API-стиля (GraphQL, gRPC, MQTT)

**Контекст**  
Система имеет трёх типов клиентов: мобильные/веб-приложения (требуют гибких запросов), внутренние сервисы (требуют высокой производительности), IoT-устройства (требуют лёгкого протокола).

**Решение**  
- **Внешние API (мобильные, веб)** – **GraphQL** для гибкости и минимизации round-trips.  
- **Внутренние сервисы** – **gRPC** для высокой производительности и поддержки стримов.  
- **IoT-устройства** – **MQTT/WebSocket** для лёгкости и двусторонней связи.

**Последствия**  
- ✅ GraphQL снижает количество запросов с клиента.  
- ✅ gRPC обеспечивает низкую задержку между сервисами.  
- ❌ Сложность поддержки трёх различных протоколов (единая документация через OpenAPI + GraphQL Schema).  

**Альтернативы**  
- Только REST – отброшен из-за избыточности запросов и неэффективности для сложных клиентов.  
- Только gRPC для внешних клиентов – отброшен из-за сложности интеграции с веб-браузерами.

### ADR-004: Выбор баз данных (Polyglot Persistence)

**Контекст**  
Разные домены имеют различные требования к хранению: транзакционная согласованность (заказы, пользователи), временные ряды (тренировки), полнотекстовый поиск (поиск людей и групп), кэширование (лидерборды, сессии).

**Решение**  
- **PostgreSQL** – для пользователей, инвентаря, промоакций (транзакционная согласованность).  
- **TimescaleDB** – для хранения временных рядов тренировок (высокая скорость вставки, агрегации).  
- **Elasticsearch** – для поиска пользователей, групп, маршрутов (полнотекстовый + гео-поиск).  
- **Redis** – для кэшей, лидербордов, онлайн-статусов (высокая производительность).  
- **S3 / MinIO** – для медиафайлов и долгосрочного хранения логов.

**Последствия**  
- ✅ Каждый сервис использует оптимальное хранилище.  
- ❌ Усложняется управление (несколько БД, разные политики бэкапов).  
- ❌ Требуется синхронизация данных между БД (например, между PostgreSQL и Elasticsearch).  

**Альтернативы**  
- Единая БД (например, PostgreSQL для всего) – отброшена из-за проблем с производительностью поиска и временных рядов.  
- NoSQL для всего – отброшена из-за необходимости транзакционной согласованности.

### ADR-005: Multi-Cloud с Kubernetes

**Контекст**  
Компания использует несколько облачных провайдеров (AWS, GCP, Azure) для снижения риска vendor lock-in и обеспечения географического резервирования.

**Решение**  
Система развёртывается в **Kubernetes** с использованием облачно-агностичных инструментов:  
- Хранилище – S3-совместимое (MinIO).  
- Очереди – Kafka (разворачивается в K8s).  
- Базы данных – StatefulSets с операторами (PostgreSQL Operator, Elasticsearch Operator).  
- Балансировка – Global Load Balancer (DNS-based).  

**Последствия**  
- ✅ Возможность миграции между облаками без переписывания кода.  
- ✅ Использование лучших предложений разных регионов.  
- ❌ Требуется экспертиза в K8s и автоматизации.  
- ❌ Усложняется управление секретами и конфигурациями (используется Vault).  

**Альтернативы**  
- Единый провайдер (например, AWS) – отброшен из-за риска vendor lock-in и недостаточной географической избыточности.

### ADR-006: Саги для распределённых транзакций

**Контекст**  
Некоторые бизнес-сценарии (например, запись тренировки и обновление инвентаря) требуют согласованности между несколькими сервисами.

**Решение**  
Используется **паттерн Saga** с координатором (оркестратором) в сервисе `Workout`. При успехе – публикация события, при ошибке – компенсирующие действия (например, откат изменения инвентаря).

**Последствия**  
- ✅ Обеспечивается согласованность данных без использования двухфазной фиксации.  
- ✅ Система остаётся отказоустойчивой (компенсации при сбоях).  
- ❌ Усложняется разработка (нужно писать компенсирующие транзакции).  
- ❌ Повышенная задержка в редких случаях (при выполнении компенсаций).  

**Альтернативы**  
- Двухфазная фиксация (2PC) – отброшена из-за снижения доступности.  
- Eventual consistency без саг – отброшена из-за бизнес-требований (нельзя допускать рассинхрон инвентаря).

### ADR-007: Аутентификация и авторизация (JWT + OAuth2)

**Контекст**  
Система должна поддерживать миллионы пользователей, включая интеграцию с социальными сетями и существующей системой аутентификации компании.

**Решение**  
- **Аутентификация** – JWT (Access Token 15 мин, Refresh Token 7 дней).  
- **Авторизация** – RBAC (роли: user, coach, admin) с проверкой через OPA.  
- **Интеграция** – OAuth2 (Google, Apple, Facebook).  

**Последствия**  
- ✅ Безопасное хранение сессий (HttpOnly, Secure, SameSite).  
- ✅ Возможность централизованного управления доступом.  
- ❌ Требуется управление ротацией ключей (используется Vault).  

**Альтернативы**  
- Сессионные cookie без JWT – отброшены из-за проблем с масштабированием (нужно хранить сессии в Redis).  
- Basic Auth – отброшен из-за небезопасности.

### ADR-008: Наблюдаемость (OpenTelemetry + ELK + Prometheus)

**Контекст**  
Распределённая система из 9+ сервисов требует комплексного мониторинга для быстрого обнаружения проблем.

**Решение**  
- **Метрики** – Prometheus + Grafana (сбор каждые 15 с).  
- **Логи** – Fluent Bit → Kafka → Logstash → Elasticsearch → Kibana.  
- **Трассировка** – OpenTelemetry + Jaeger (сбор спанов, хранение в Elasticsearch).  

**Последствия**  
- ✅ Полная наблюдаемость сквозных запросов.  
- ✅ Возможность построения бизнес-метрик (успешные заказы, время оформления).  
- ❌ Высокая стоимость хранения (до 3 ТБ логов в сутки).  
- ❌ Сложность настройки (требуется компетенция в ELK и OpenTelemetry).  

**Альтернативы**  
- Только логи – отброшены (недостаточно для поиска причин сбоев).  
- Splunk – отброшен из-за высокой стоимости.
